# Platform detection & version gates

Use this as the lookup table during Phase 1 (Discovery + version gating). Stop and warn the user if any **gate** below is violated ‚Äî installing the SDK on an unsupported platform will fail at build time.

## 1. Primary-platform detection

Run all checks; the **most specific** match wins.

| Signal (file or content) | Implies |
|---|---|
| `package.json` with a `react-native` dep, no `expo` dep | Bare React Native |
| `package.json` with both `expo` and `react-native` deps; `app.json` / `app.config.{js,ts}` present | Expo |
| `Package.swift` at repo root, no Android sources | Pure SPM iOS / macOS package |
| `*.xcodeproj` or `*.xcworkspace` + `Podfile` | iOS app, CocoaPods |
| `*.xcodeproj` or `*.xcworkspace`, no `Podfile` | iOS app, SPM |
| `build.gradle.kts` or `build.gradle` + `AndroidManifest.xml`, no `commonMain/` source set | Pure Android (Java/Kotlin) |
| `build.gradle.kts` + `kotlin { ... }` block referencing `android()`, `iosX64()`, `iosArm64()`, etc., AND a `commonMain/` source set | Kotlin Multiplatform (KMP) |
| Hybrid: KMP + a separate iOS Xcode project consuming a generated `.xcframework` | KMP with native UI shells (treat each shell separately for SDK install ‚Äî each app target needs its own initialisation) |

If both Android (`/android`) and iOS (`/ios`) folders exist as siblings of `package.json`, you are looking at React Native or Expo ‚Äî handle through the JS package manager, not the native ones.

## 2. RN flavour disambiguation ‚Äî **four**, not two

This is the most error-prone discovery step in the entire skill. Get it wrong and you either (a) wipe the user's hand-managed `ios/` / `android/` folders by recommending `expo prebuild`, or (b) add a config-plugin entry that does nothing.

### The four flavours

| Flavour | `react-native` in `package.json` | `expo` in `package.json` | `ios/` & `android/` in git | `app.json` plugins active? | Run `prebuild`? |
|---|---|---|---|---|---|
| **A. Pure Bare RN** | yes | no | yes (hand-managed) | n/a (no app.json or unused) | ‚ùå never |
| **B. Bare RN + Expo Modules** | yes | yes (`expo-modules-core` + specific `expo-*` packages) | **yes (hand-managed)** | ‚ùå ignored ‚Äî no prebuild ever runs | **‚ùå never ‚Äî would wipe `ios/` and `android/`** |
| **C. Expo Prebuild Workflow** (a.k.a. "Expo Bare") | yes | yes | **no ‚Äî generated by `prebuild`** (often `.gitignore`d) | ‚úÖ yes ‚Äî `prebuild` regenerates native folders from these | ‚úÖ yes (`expo prebuild --clean`) |
| **D. Expo Managed** | no | yes | no (never present) | ‚úÖ yes | n/a (CNG, EAS) |

### How to actually tell B from C

The key is **whether `ios/` and `android/` are committed to git as hand-managed source**:

```bash
# Run inside the project root.
git ls-files ios/ android/ | head -3
```

- If the command lists hand-edited `AppDelegate.swift`, `MainApplication.kt`, custom `Podfile`, native modules under `ios/MyProject/` or `android/app/src/main/java/...` ‚Äî that's **Flavour B** (Bare + Expo Modules). The native folders are the source of truth. **Never** run `prebuild`.
- If `ios/` and `android/` are absent, `.gitignore`d, or only contain trivial generated stubs (no hand-edited Swift/Kotlin) ‚Äî that's **Flavour C** (Expo Prebuild Workflow). `prebuild --clean` is the correct command.

Other Flavour-B tells:
- `package.json` has `"expo-modules-core"` plus a handful of `expo-*` deps (e.g. `expo-image-picker`, `expo-secure-store`).
- `app.json` exists but its `"plugins": [...]` array contains Expo Modules plugins (`expo-font`, `expo-image-picker`, etc.) ‚Äî those are real but **none of them trigger `prebuild`**, they only document permissions / fonts the project's hand-managed native code already wires.
- There's a `Podfile` in `ios/` with hand-rolled `use_expo_modules!` and manually maintained pod entries.

### Sub-case: Flavour B + project-authored Expo Modules

Some Flavour-B projects ship their own Expo Modules (e.g. `modules/expo-foo/` with native iOS/Android sources registered via `expo-module.config.json`). This adds wiring on top of Flavour B but **does not change the Amply integration path**:

- Amply is not an Expo Module ‚Äî install via standard `yarn add` + Expo Modules autolinking.
- Do not run `prebuild` ‚Äî same rule as base Flavour B.
- The user's custom Expo Modules continue to be picked up by `expo-modules-autolinking` during `yarn expo run:ios` / `run:android` alongside the Amply package.

Treat this sub-case identically to Flavour B for everything in this skill.

### Consequences for the Amply integration

| Flavour | Add `@amplytools/...` to `app.json` plugins? | Native registration | iOS install | Android install |
|---|---|---|---|---|
| A. Pure Bare RN | ‚ùå no ‚Äî there's no plugin runner | RN autolinking via `react-native.config.js` (automatic on RN ‚â• 0.79) | `cd ios && pod install` | Gradle sync |
| B. Bare RN + Expo Modules | ‚ùå **no** ‚Äî entry would be a no-op and misleading; the plugin never runs | RN autolinking (same as A); Expo Modules autolinking runs in parallel but doesn't trigger Amply's plugin | `cd ios && pod install` | Gradle sync |
| C. Expo Prebuild Workflow | ‚úÖ **yes** ‚Äî required; the plugin mutates `MainApplication.kt` during `prebuild` | The plugin registers `AmplyPackage`; `prebuild` writes it into generated code | `npx expo prebuild --clean` then `pod install` | `npx expo prebuild --clean` then Gradle sync |
| D. Expo Managed | ‚úÖ yes ‚Äî same as C; built remotely by EAS | EAS Build runs the plugin during the cloud build | `eas build -p ios` | `eas build -p android` |

For Android specifically: if the project is Flavour A or B and **autolinking fails to register the package** (rare on RN ‚â• 0.79 but possible on older RN with mismatched codegen), you may need to manually edit `MainApplication.kt`:

```kotlin
override fun getPackages(): List<ReactPackage> =
    PackageList(this).packages.apply {
        // add(AmplyPackage())   // only if autolinking didn't already register it
    }
```

Don't add this preemptively ‚Äî only after confirming autolinking missed.

## 3. Package manager detection

| File present | Use |
|---|---|
| `yarn.lock` | yarn |
| `package-lock.json` | npm |
| `pnpm-lock.yaml` | pnpm |
| `bun.lockb` | bun |
| `Podfile.lock` | CocoaPods (iOS) |
| `Package.resolved` and no `Podfile` | Swift Package Manager (iOS) |
| Both `Podfile` and `Package.resolved` | Mixed ‚Äî prefer the one the existing AmplySDK consumer would expect (CocoaPods or SPM both supported). |
| `gradle/wrapper/gradle-wrapper.properties` | Gradle (Android / KMP) |
| `settings.gradle.kts` with `dependencyResolutionManagement` | Modern Gradle settings ‚Äî KMP-friendly |

## 4. Navigation-library detection

Grep imports / dependencies:

| Library | Import / dep marker |
|---|---|
| React Navigation | `@react-navigation/native`, `@react-navigation/stack` |
| expo-router | `expo-router` dep, `app/` directory with file-based routes |
| Reanimated / Stack mixed | `react-native-screens` + `@react-navigation/*` |
| SwiftUI NavigationStack | Swift files importing `SwiftUI` with `NavigationStack { ... }` |
| UIKit | `AppDelegate.swift` with `UIApplicationDelegate` and no SwiftUI `App` struct |
| Compose Navigation | `androidx.navigation:navigation-compose` dep, `NavHost` calls |
| Jetpack Navigation (XML) | `androidx.navigation:navigation-fragment-ktx`, `nav_graph.xml` |
| Voyager (Compose) | `cafe.adriel.voyager:voyager-navigator` |
| Decompose (KMP) | `com.arkivanov.decompose:decompose` |

## 5. Version gates

Stop and warn the user when any of these fail:

| Gate | Pass condition | Source |
|---|---|---|
| React Native version | `>= 0.79` AND New Architecture enabled (`newArchEnabled=true` in `gradle.properties` and `RCT_NEW_ARCH_ENABLED=1` for iOS) | RN SDK README |
| Expo SDK | `>= 53` | RN SDK README |
| Android `minSdk` (RN target) | `>= 24` | RN SDK README |
| Android `minSdk` (KMP target) | `>= 21` | KMP capabilities reference |
| iOS deployment target (RN) | `>= 15.1` | `AmplyReactNative.podspec` shipped with `@amplytools/react-native-amply-sdk@0.2.x` declares `platforms = { :ios => '15.1' }`. The RN SDK README still says 13.0+ but `pod install` fails below 15.1. Verify against the installed package's podspec. |
| iOS deployment target (KMP / native) | `>= 14.1` | KMP capabilities reference |
| Kotlin compiler | KMP-compatible Kotlin (1.9+) for KMP projects | Inferred from KMP samples |

**Amply SDK floor (capability axis, not a stop-gate):** alongside the host-platform gates above, record which **Amply SDK version** the project pins. Event-history audience targeting (event counts, first/last-occurrence dates, event property filters) only matches apps running **Amply SDK 0.6.1+** ‚Äî devices on older SDK builds don't fail, they simply never match campaigns that use event conditions. Pin `0.6.1` or later on new integrations (the install snippets in every `sdk-cheatsheet-*.md` do); for an existing integration, note the pinned version in the audit so the team knows which targeting vocabulary their fleet supports.

**How to check:**

- React Native: `npx react-native --version` or read `package.json` ‚Üí `react-native`.
- Expo: `package.json` ‚Üí `expo`.
- Android `minSdk`: `android/app/build.gradle` (RN) or `library/build.gradle.kts` (KMP).
- iOS deployment target: top of `*.pbxproj` (`IPHONEOS_DEPLOYMENT_TARGET`) or `Package.swift` (`platforms: [.iOS(.v14)]`).
- New Architecture (Android): `android/gradle.properties` ‚Üí `newArchEnabled=true`.
- New Architecture (iOS): check in this order ‚Äî `ios/Podfile.properties.json` ‚Üí `newArchEnabled`; top of `ios/Podfile` for `ENV['RCT_NEW_ARCH_ENABLED']`; project's `package.json` `ios` script for an explicit `RCT_NEW_ARCH_ENABLED=1`. On RN ‚â• 0.79 the default is on, but projects can opt out by setting `0` ‚Äî verify before assuming.
